# keenetic-vpn-mihomo

VPN на роутере **Keenetic** для выбранных устройств. Один скрипт ставит
[mihomo](https://github.com/MetaCubeX/mihomo) (clash.meta) с веб-интерфейсом,
через VPN ходят только те устройства, которые вы выбрали — мышкой в админке
Keenetic или командой в консоли. Остальная сеть работает напрямую.

Принимает обычные happ-style **VLESS-подписки**: вставил URL — получил список
серверов; сервер и подписку можно менять в любой момент.

## Возможности

- **Выбор устройств в админке Keenetic**: установщик создаёт политику доступа
  «te-vpn» — назначаете её устройству, и оно ходит через VPN. Или из консоли:
  `te-vpn add <mac|ip>`.
- **Подписки**: `te-vpn sub "<url>"` — mihomo сам тянет и обновляет список
  серверов.
- **Выбор сервера**: автоматически по health-check (`te-vpn auto`), вручную
  (`te-vpn select`) или мышкой в веб-интерфейсе.
- **Исключения из VPN**: сервисы из
  `/opt/etc/mihomo/rules/direct-services.yaml` всегда идут напрямую
  (`DIRECT`) даже для устройств, выбранных в VPN.
- **Предсказуемое восстановление**: при управляемой остановке правила снимаются
  и устройства идут напрямую; при сбоях monit быстро поднимает mihomo обратно
  (см. [Если VPN не работает](#если-vpn-не-работает)).
- **Живучесть**: автозапуск при загрузке, monit-супервизия, переживает
  переподключение WAN и пересборку файрвола прошивкой.
- **Веб-интерфейс**: metacubexd на `http://<ip-роутера>:9090/ui` — статус,
  замеры задержек, переключение серверов, живой трафик.

## Установка

Запускается с рабочей машины, на роутер ничего клонировать не нужно — скрипт
всё делает по SSH (по умолчанию `root@192.168.1.1:222`, переопределяется
`ROUTER=… PORT=…`). На роутере нужен только Entware.

```sh
sh scripts/install.sh
ssh root@192.168.1.1 -p 222 'te-vpn sub "https://your-subscription-url"'
```

Скрипт сам: ставит зависимости через opkg (curl, iptables, monit),
определяет архитектуру CPU и качает нужный бинарник mihomo с GitHub-релизов,
раскладывает файлы, создаёт политику доступа «te-vpn», запускает и отдаёт под
супервизию. Осталось выбрать устройства — см. следующий раздел.

Повторный запуск безопасен и работает как **обновление**: живые
`te-vpn.conf` / `devices.list` не затираются (если репо-версия отличается,
она ляжет рядом как `*.new`). `config.yaml` сохраняет подписку и локальные настройки, но установщик
идемпотентно приводит технические поля transparent proxy к актуальной схеме
(`redir-port`, `tproxy-port`, sniffer, DNS mode, disabled TUN), предварительно
сохранив backup рядом.

Флаги установки:

| Переменная | Что делает |
|---|---|
| `ROUTER=root@10.0.0.1 PORT=22` | другой роутер |
| `POLICY_NAME=MyVPN` | своё имя политики доступа |
| `WAN_IF=PPPoE0` | запасное имя WAN для политики, если автоопределение подключений ничего не нашло |
| `NO_POLICY=1` | не создавать политику — управление только через `te-vpn add` |
| `NO_MONIT=1` | без monit-супервизии |
| `NO_START=1` | только разложить файлы, не запускать |

Удаление: `sh scripts/uninstall.sh` — снимает роутинг, удаляет политику и
файлы; конфиг с подпиской сохраняется (`PURGE=1` — снести всё).

## Выбор устройств: кто ходит через VPN

Два способа, работают одновременно.

**Через веб-админку** (политика доступа «te-vpn», создана установщиком):

1. Админка → **«Мои сети и Wi-Fi» → «Список устройств»**, зарегистрируйте
   устройство, если оно ещё не зарегистрировано.
2. В карточке устройства в поле **«Политика доступа в интернет»** выберите
   **te-vpn**. Готово.

Убрать из VPN — вернуть устройству политику по умолчанию. Сами политики
живут в «Сетевые правила» → «Приоритеты подключений», менять там ничего не
нужно.

**Через консоль** — для устройств, которых нет в списке админки, или целых
подсетей:

```sh
te-vpn add aa:bb:cc:dd:ee:ff laptop    # по MAC
te-vpn add 192.168.1.50 tv             # по IP
te-vpn add 192.168.1.64/28 iot         # по подсети
te-vpn remove aa:bb:cc:dd:ee:ff
te-vpn devices
```

Уже есть своя политика? Впишите её имя в `POLICY_NAME=` в
`/opt/etc/mihomo/te-vpn.conf` — te-vpn найдёт её метку сам.

## Команды

```sh
te-vpn status                 # процесс, transparent proxy, устройства, роутинг, сервер, подписка
te-vpn sub "https://..."      # задать/сменить подписку
te-vpn servers                # список серверов из подписки
te-vpn auto                   # автоматический выбор сервера (рекомендуется)
te-vpn select "<name>"        # закрепить конкретный сервер
te-vpn flush                  # сбросить старые соединения выбранных устройств
te-vpn reload                 # перечитать конфиг + обновить подписку
te-vpn add / remove / devices # устройства (см. выше)
te-vpn start / stop / restart
te-vpn log [N]
```

То же самое мышкой: веб-интерфейс на `http://<ip-роутера>:9090/ui`.

## Если VPN не работает

Нормальный фоллбек при управляемой остановке — «напрямую, без VPN», а monit
быстро поднимает mihomo после сбоев:

- **Умер процесс mihomo** (креш, не стартует). monit перезапускает процесс и
  заново применяет transparent proxy-правила. До рестарта возможна короткая
  пауза для выбранных устройств, потому что TCP продолжает редиректиться в
  локальный порт mihomo. Управляемая остановка (`te-vpn stop/down`) снимает
  правила сразу, и устройства возвращаются в обычную маршрутизацию.
- **VPN-сервер не работает** (выключен, протух ключ, подключается, но не
  пропускает трафик). Группа `AUTO` каждые 2 минуты проверяет серверы
  реальным запросом и выбирает живую VPN-ноду из подписки. `DIRECT` намеренно
  не входит в `AUTO`, чтобы режим «VPN включён» не превращался молча в обычный
  интернет. Если нужно временно отключить VPN для выбранных устройств, выберите
  `PROXY -> DIRECT` в веб-интерфейсе.
- **Подписка ещё не задана / не скачалась.** mihomo стартует, но при выборе
  `AUTO` трафику будет некуда идти, пока не появится хотя бы один сервер.
  Для временного прямого выхода выберите `PROXY -> DIRECT`.

Единственный режим без страховки — вручную закреплённый сервер
(`te-vpn select`): если умер именно он, mihomo будет слать трафик в него,
пока не переключитесь (`te-vpn auto`). Поэтому рекомендуемый режим — `AUTO`.

Если правила выглядят правильно, но конкретное устройство всё ещё “едет” старым
маршрутом после смены политики/сервера, сбросьте его старые conntrack-потоки:
`te-vpn flush`. Скрипт делает это автоматически при `te-vpn reload`,
`te-vpn select`, `te-vpn auto`, изменении списка устройств, WAN-событиях и
пересборке firewall прошивкой. Повторные NDM-вызовы сглаживаются
`CONNTRACK_FLUSH_INTERVAL`, чтобы не рвать соединения дважды подряд.

## Как это работает

VPN-трафик отбирается по fwmark, а доставляется в mihomo прозрачным прокси:

1. Устройства из **политики доступа** метит сама прошивка: у каждой политики
   Keenetic есть свой connmark, `te-vpn` находит его по имени политики
   (`ndmc -c "show ip policy"`). Устройства из **`devices.list`** метит
   собственный chain в `mangle/PREROUTING` (по MAC или IP) fwmark'ом `0x6d69`.
2. TCP выбранных устройств цепочка `MIHOMO_REDIR` в `nat/PREROUTING`
   отправляет в `redir-port: 7892`. Это стабильнее на Keenetic, чем TCP через
   TPROXY.
3. UDP выбранных устройств цепочка `MIHOMO_TP` в `mangle/PREROUTING`
   отправляет в `tproxy-port: 7895`; для локальной доставки используется
   `ip rule fwmark 0x1/0x1 lookup 1000 priority 99` и таблица 1000 с
   `local default dev lo`.
4. QUIC/HTTP3 (`udp/443`) для VPN-устройств по умолчанию разрешён и проходит
   через UDP TPROXY. Если конкретный сервер плохо релеит UDP/443, можно вернуть
   старое поведение через `BLOCK_QUIC=1` в `/opt/etc/mihomo/te-vpn.conf`, чтобы
   приложения откатывались на обычный TCP/TLS через VPN.
5. Sniffer в mihomo восстанавливает HTTP Host / TLS SNI, поэтому в веб-морде
   видны домены, а приложения получают более предсказуемую прозрачную
   проксификацию. Непомеченный трафик ходит напрямую.
6. Перед финальным правилом `MATCH,PROXY` применяется локальный список
   `direct-services`: домены из `/opt/etc/mihomo/rules/direct-services.yaml`
   получают `DIRECT`. Так можно оставить приставку в VPN-политике, но пустить
   Кинопоиск, VK, ivi, Okko и другие чувствительные к VPN российские
   медиа-сервисы через обычный интернет.

NDM-хуки держат это живым: `netfilter.d` восстанавливает transparent-proxy
цепочки и локальное TPROXY-правило после пересборки файрвола прошивкой,
`wan.d` повторно применяет роутинг после переподключения WAN. Метка политики
Keenetic кэшируется в
`/opt/var/run/te-vpn.policy_mark`, чтобы во время пересборки файрвола
устройства из веб-политики не выпадали из transparent proxy, если `ndmc`
временно не отдаёт список политик.

Параметры (fwmark, имя политики, redir/tproxy-порты, таблица, приоритет,
QUIC/conntrack-поведение) — в `/opt/etc/mihomo/te-vpn.conf`.

## Структура репозитория

Дерево `router/` зеркалит `/opt` на роутере — файл лежит в репо там же, куда
он попадёт при деплое:

| Файл (`router/…` → `/opt/…`) | Что это |
|---|---|
| `bin/te-vpn` | вся логика: CLI, запуск/остановка, маркировка, роутинг |
| `etc/mihomo/config.yaml` | конфиг mihomo: подписка, группы, transparent proxy, DNS |
| `etc/mihomo/te-vpn.conf` | настройки te-vpn: fwmark, имя политики, redir/tproxy-порты, таблица |
| `etc/mihomo/devices.list` | устройства, добавленные через `te-vpn add` |
| `etc/mihomo/rules/direct-services.yaml` | локальный список доменов, которые всегда идут напрямую |
| `etc/init.d/S06mihomo` | автозапуск при загрузке (шим над te-vpn) |
| `etc/monit.d/mihomo.conf` | monit-супервизия (зовёт te-vpn) |
| `etc/ndm/wan.d/10-mihomo.sh` | восстановление роутинга при WAN up |
| `etc/ndm/netfilter.d/50-mihomo.sh` | восстановление mangle/nat-цепочек после пересборки файрвола |

Бинарник mihomo (`/opt/sbin/mihomo`) и веб-UI (`/opt/etc/mihomo/ui`) в
репо не хранятся — install-скрипт скачивает их на роутер с GitHub-релизов.
`scripts/` — установка (`install.sh`) и удаление (`uninstall.sh`) по SSH.

## Требования

- Роутер Keenetic с Entware в `/opt` и SSH-доступом из LAN (политика доступа
  требует прошивку с `ndmc`; без него — режим `devices.list`)
- Архитектуры: aarch64 / armv7 / armv5 / mips(el) / x86_64 — определяется
  автоматически; версия mihomo задаётся в `scripts/install.sh`
- Проверено на Keenetic NC-2312 (MT7981B, aarch64, kernel 4.9-ndm-5)
